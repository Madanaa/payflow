# PayFlow

PayFlow is a Spring Boot REST API for registering wallet users and recording UPI-style money transfers. It uses Spring Web MVC, Spring Data JPA, and an in-memory H2 database.

## How to Run

```powershell
cd Pay-Flow
.\mvnw.cmd spring-boot:run
```

API base URL:

```text
http://localhost:8080
```

H2 console:

```text
http://localhost:8080/h2-console
JDBC URL: jdbc:h2:mem:payflowdb
User: sa
Password: empty
```

Run tests:

```powershell
cd Pay-Flow
.\mvnw.cmd test
```

## Project Layers

`entity` contains the JPA classes that become database tables: `User` and `Transaction`.

`repository` contains Spring Data JPA interfaces. `JpaRepository` provides methods like `save`, `findAll`, and `findById`.

`service` contains business methods such as `registerUser`, `findByUpiId`, and `sendMoney`.

`controller` contains REST endpoints. It receives HTTP requests, maps JSON into Java objects, calls services, and returns JSON responses.

## Spring Boot Features

Embedded server: the project runs directly with `spring-boot:run`; no separate Tomcat setup is required.

Auto-configuration: Spring Boot configures Spring MVC, H2, JPA, repository scanning, JSON converters, and the H2 console from the dependencies and `application.properties`.

Production-ready defaults: the app gets logging, dependency management, sensible error responses, and default runtime configuration without XML setup.

## Generated Tables

Hibernate creates these tables automatically:

```sql
create table users (
    user_id bigint generated by default as identity,
    balance float(53),
    name varchar(255),
    phone_number varchar(255),
    upi_id varchar(255),
    primary key (user_id)
)
```

```sql
create table transactions (
    transaction_id bigint not null,
    amount float(53),
    note varchar(255),
    receiver_upi_id varchar(255),
    sender_upi_id varchar(255),
    primary key (transaction_id)
)
```

H2 queries:

```sql
SELECT * FROM USERS;
SELECT * FROM TRANSACTIONS;
```

## REST Endpoints

### Register User

```powershell
curl.exe --% -X POST http://localhost:8080/users -H "Content-Type: application/json" -d "{\"name\":\"Madan\",\"upiId\":\"madan@okaxis\",\"balance\":5000,\"phoneNumber\":\"9876543210\"}"
```

Example response:

```json
{"name":"Madan","upiId":"madan@okaxis","balance":5000.0,"phoneNumber":"9876543210","userId":1}
```

### List Users

```powershell
curl.exe -X GET http://localhost:8080/users
```

### Get User by ID

```powershell
curl.exe -X GET http://localhost:8080/users/1
```

### Get User by UPI ID

```powershell
curl.exe -X GET http://localhost:8080/users/upi/madan@okaxis
```

### Send Money

```powershell
curl.exe --% -X POST http://localhost:8080/transactions -H "Content-Type: application/json" -d "{\"senderUpiId\":\"madan@okaxis\",\"receiverUpiId\":\"rahul@ybl\",\"amount\":500,\"note\":\"Test payment\"}"
```

Example response:

```json
{"transactionId":1,"senderUpiId":"madan@okaxis","receiverUpiId":"rahul@ybl","amount":500.0,"note":"Test payment"}
```

### Users With Balance Above Amount

```powershell
curl.exe -X GET http://localhost:8080/users/balance-above/4000
```

## Repository Queries

Simple lookup:

```java
Optional<User> findFirstByUpiIdOrderByUserIdAsc(String upiId);
```

This query is derived from the method name. `findFirstByUpiId` tells Spring Data JPA to search by the `upiId` field and return one matching record. `OrderByUserIdAsc` makes the result deterministic if old duplicate data exists.

Duplicate check:

```java
boolean existsByUpiId(String upiId);
```

JPQL custom query:

```java
@Query("SELECT u FROM User u WHERE u.balance > :minBalance")
List<User> findUsersWithBalanceGreaterThan(@Param("minBalance") Double minBalance);
```

Derived query methods are best for simple lookups. JPQL with `@Query` is useful when the query is more custom but should still use entity names and Java field names. Native SQL is least preferred because it depends directly on database-specific table names, column names, and SQL syntax.

## RequestBody Demo

With `@RequestBody`, Spring uses Jackson to convert JSON into a populated `User` object:

```text
User{userId=null, name='Madan', upiId='madan@okaxis', balance=5000.0, phoneNumber='9876543210'}
```

Without `@RequestBody`, Spring MVC does not read the raw JSON request body into the object. It tries to bind from request parameters/form fields instead, so the fields stay null.

## Conceptual Write-Up

Request lifecycle: When `curl` sends `POST /users`, the embedded server receives the request and passes it to Spring MVC. The Dispatcher Servlet acts as the front controller and finds the matching controller method. A Handler Adapter invokes `createUser`, then Spring serializes the returned Java object into JSON.

Serialization: Jackson converts JSON into Java objects through Spring's HTTP message converters. JSON keys like `name` and `upiId` match Java bean properties. A key like `upi_id` will not map to `upiId` by default unless a naming strategy or `@JsonProperty` is configured.

Spring vs Spring Boot: Plain Spring would require manual server setup, Dispatcher Servlet configuration, Jackson converters, JPA setup, datasource setup, and repository scanning. Spring Boot handles those automatically through starters and auto-configuration.

Stateless REST: Each request carries all the data needed to process it. This matters when the app runs on multiple servers because any server can handle any request as long as shared state is stored in the database.

Persistence: A Java `List` would lose all transaction data when the server restarts. A payment app must persist records in a database so payment history remains available and auditable.
