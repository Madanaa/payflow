# PayFlow

PayFlow is a Spring Boot REST API for registering wallet users and recording UPI-style money transfers. It uses Spring Web MVC, Spring Data JPA, and an in-memory H2 database.

## How to Run

```powershell
.\mvnw.cmd spring-boot:run
```

The API starts at:

```text
http://localhost:8080
```

The H2 console is available at:

```text
http://localhost:8080/h2-console
JDBC URL: jdbc:h2:mem:payflowdb
User: sa
Password: empty
```

Run tests with:

```powershell
.\mvnw.cmd test
```

## Layers

`entity` contains the JPA classes that become database tables: `User` and `Transaction`.

`repository` contains Spring Data JPA interfaces. `JpaRepository` gives CRUD methods such as `save`, `findAll`, and `findById` without handwritten SQL.

`service` contains business methods such as `registerUser`, `findByUpiId`, and `sendMoney`. Controllers call services instead of calling repositories directly.

`controller` contains REST endpoints. It receives HTTP requests, maps JSON into Java objects, calls services, and returns JSON responses.

## Spring Boot Features in This Project

Embedded server: the project runs with `.\mvnw.cmd spring-boot:run`; no separate Tomcat install is needed because Spring Boot starts the embedded server.

Auto-configuration: Spring Boot sees Spring Web MVC, Spring Data JPA, and H2 on the classpath, then configures the web layer, `DataSource`, JPA `EntityManagerFactory`, repositories, and H2 console.

Production-ready defaults: the app has sensible logging, JSON conversion, error responses, dependency versions, and health-oriented startup behavior without manual XML configuration.

## Entities and Generated Tables

Hibernate generated these table statements on startup:

```sql
create table transactions (
    transaction_id bigint not null,
    amount float(53),
    note varchar(255),
    receiver_upi_id varchar(255),
    sender_upi_id varchar(255),
    primary key (transaction_id)
)
```

```sql
create table users (
    user_id bigint generated by default as identity,
    balance float(53),
    name varchar(255),
    phone_number varchar(255),
    upi_id varchar(255),
    primary key (user_id)
)
```

The project uses table names `users` and `transactions` because `USER` and `TRANSACTION` are awkward/reserved names in H2 SQL. In the H2 console, use:

```sql
SELECT * FROM USERS;
SELECT * FROM TRANSACTIONS;
```

Before inserting data, both tables should show the correct columns and no rows. After the sample curl calls below, `USERS` contains Priya and Rahul, and `TRANSACTIONS` contains one transfer from `priya@okaxis` to `rahul@ybl`.

## Repository Queries

`UserRepository` has this derived query:

```java
Optional<User> findByUpiId(String upiId);
```

When `GET /users/upi/priya@okaxis` runs, Hibernate logs:

```sql
select
    u1_0.user_id,
    u1_0.balance,
    u1_0.name,
    u1_0.phone_number,
    u1_0.upi_id
from
    users u1_0
where
    u1_0.upi_id=?
```

JPA derives the query from the method name: `findBy` means create a select query, and `UpiId` maps to the `upiId` Java field, which is stored as the `upi_id` column. The `?` is a parameter placeholder; Hibernate safely binds the method argument, for example `priya@okaxis`, into that placeholder.

The JPQL custom query is:

```java
@Query("SELECT u FROM User u WHERE u.balance > :minBalance")
List<User> findUsersWithBalanceGreaterThan(@Param("minBalance") Double minBalance);
```

Derived method names are best for simple conditions like `findByUpiId`. `@Query` with JPQL is better when the query is more explicit but should still use entity names and Java field names. Native SQL is least preferred because it ties the code to one database's table names, column names, and SQL dialect, making refactoring and database changes harder.

## Endpoints

### Register User

Windows PowerShell curl:

```powershell
curl.exe --% -i -s -X POST http://localhost:8080/users -H "Content-Type: application/json" -d "{\"name\":\"Priya\",\"upiId\":\"priya@okaxis\",\"balance\":5000,\"phoneNumber\":\"9876543210\"}"
```

Output:

```json
{"name":"Priya","upiId":"priya@okaxis","balance":5000.0,"phoneNumber":"9876543210","userId":1}
```

Second user:

```powershell
curl.exe --% -i -s -X POST http://localhost:8080/users -H "Content-Type: application/json" -d "{\"name\":\"Rahul\",\"upiId\":\"rahul@ybl\",\"balance\":3000,\"phoneNumber\":\"9123456780\"}"
```

Output:

```json
{"name":"Rahul","upiId":"rahul@ybl","balance":3000.0,"phoneNumber":"9123456780","userId":2}
```

## RequestBody Demonstration

With `@RequestBody`, the controller log shows the JSON body converted into a populated object:

```text
Incoming request to save user: User{userId=null, name='Priya', upiId='priya@okaxis', balance=5000.0, phoneNumber='9876543210'}
```

Without `@RequestBody`, this demo endpoint receives the same JSON body but the object fields stay null:

```powershell
curl.exe --% -s -X POST http://localhost:8080/users/demo/without-request-body -H "Content-Type: application/json" -d "{\"name\":\"NoBody\",\"upiId\":\"nobody@upi\",\"balance\":1}"
```

Output:

```json
{"name":null,"upiId":null,"balance":null,"phoneNumber":null,"userId":null}
```

The fields are null without `@RequestBody` because Spring MVC does not read the raw JSON request body into the method parameter. Without that annotation, it treats the `User` as a model-style object and tries to bind values from request parameters or form fields. Since the JSON values are only inside the request body, no matching request parameters exist, so the Java object is created with default null fields.

## Conceptual Write-Up

Request lifecycle: When `curl` sends `POST /users`, the embedded server receives the HTTP request and passes it into Spring MVC. The Dispatcher Servlet is the front controller; it looks for a matching controller method using the request path and HTTP method. A Handler Adapter then calls the matched `createUser` method in `UserController`. After the method returns, Spring serializes the returned `User` object into JSON for the HTTP response.

Serialisation: Jackson, through Spring's HTTP message converters, converts JSON such as `{"name":"Priya","upiId":"priya@okaxis"}` into a Java `User` object. It matches JSON property names to Java bean properties such as `name` and `upiId`. If the JSON key is `upi_id` instead of `upiId`, it will not match this Java property by default in this project, so `upiId` can remain null unless a naming strategy or `@JsonProperty("upi_id")` is configured.

Spring Boot features: The three features are embedded server, auto-configuration, and production-ready defaults. The embedded server runs the REST API on port `8080` without installing Tomcat separately. Auto-configuration creates MVC, H2, JPA, and repository infrastructure from the dependencies and `application.properties`. Production-ready defaults give usable logging, JSON error responses, dependency management, and sensible runtime behavior.

Spring vs. Spring Boot: With plain Spring, PayFlow would need manual setup for the web server, Dispatcher Servlet, component scanning, Jackson converters, JPA configuration, datasource configuration, repository scanning, and dependency versions. Spring Boot supplies these through starters and auto-configuration. That lets this project focus on entities, repositories, services, and controllers instead of infrastructure code.

Stateless REST: Stateless means each HTTP request contains everything the server needs to process it; the endpoint does not depend on remembering a previous request in server memory. `POST /transactions` reads the JSON body, saves one transaction, and returns the saved record. This matters behind a load balancer because request one might go to server A and request two might go to server B. Shared state belongs in the database, not in one server's memory.

Persistence: If transactions were stored only in a Java `List`, all records would disappear when the server restarted because the list lives only in process memory. H2 stores the records through the database layer while the app is running, and a real payments system would use a durable external database. Losing transactions on restart is unacceptable because payment history is financial evidence and must remain consistent, auditable, and recoverable.
